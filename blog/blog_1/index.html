GBB method: Toy example

Introduction

High-resolution functional magnetic resonance imaging (fMRI) at ultra-high magnetic field strengths of 7 Tesla and above extends the possibilities to study brain functioning in the living human brain. One of the most exciting purposes of high-resolution fMRI is the differentiation of brain activation within the cerebral cortex. This information is especially valuable since it is known that most of the cortex is functionally organized into so called cortical layers and columns.

However, to study brain activation within the cortex, we have to know the exact location of the cortex. This is not trivial because functional images are pushed in their sensitivity to detect neurally induced changes and not in their contrast between gray matter (GM), white matter (WM), and cerebrospinal fluid (CSF).

Therefore — as for more conventional fMRI applications — we still rely on segmentations which are taken from separate anatomical scans. This comes along with all usual registration problems. The slight difference to conventional fMRI applications is that we can start seeing these anatomical properties in the functional images as well. To base an entire segmentation on those images might still be quite tricky, but we can easily use this additional information to improve an already existing segmentation.

In this short blog post, I will introduce a method that exactly tries to achieve this. I illustrate the method on a toy example and might show an example using actual brain data in a future blog post.

Definition of the toy example

The toy example consists of a brain which has the shape of a cube and is shown in cross section in fig. 1. More precisely, the brain is defined as an image array with matrix size $[320,320,240]$ and voxel resolution $0.7\times0.7\times0.7\text{ mm}^3$. WM is located in the center of the array and is completely surrounded by GM. The green sphere within WM is the computed GM/WM boundary (source mesh) from a hypothetical different brain scan. Apparently, there is a large mismatch between computed GM/WM boundary (sphere) and actual GM/WM boundary (cube). This might have been caused by segmentation or registration inaccuracies. The aim is now to deform the initial surface mesh to match the actual GM/WM boundary, which I then refer to as gradient-based boundary (GBB).

Gradient-based boundary (GBB)

The current implementation of the GBB method can be found here. In this section, I will briefly introduce the simple idea behind it. We start by randomly selecting a vertex from the source mesh. The vertex shall be moved to the nearest GM/WM boundary found in the functional image. Boundaries are locations with high contrast and are defined as locations where the image gradient gets maximal. It is now checked in which direction we have to move the vertex. Movements are only allowed along one defined dimension which can be either the surface normal or one coordinate axis of the image array. For functional imaging, the latter might be beneficial because we can then restrict all movements to be along the phase-encoding direction which is most affected by distortions in echo-planar imaging. The vertex plus a defined neighborhood is moved in the chosen direction with a specific learning rate (size of movement). Multiple neighborhoods and learning rates can be used to allow finer adjustments at the end. New vertices are selected iteratively until the used cost function is below a defined threshold. I decided to apply the cost function that is used in the boundary-based registration method [1]. For the toy example, the following parameters were set:

reg_params["t2s"] = True # GM: bright, WM: dark
reg_params["line_dir"] = 3 # line along surface normals
reg_params["line_length"] = 10 # line length in mm
reg_params["r_size"] = [5, 2.5, 1] # neighborhood size
reg_params["l_rate"] = [0.5, 0.25, 0.1] # learning rate
reg_params["max_iter"] = [250000, 500000, 1000000] # maximum number of iterations
reg_params["cost_threshold"] = [1e-7, 1e-8, 1e-9] # cost function threshold
reg_params["gradient_sigma"] = 1 # gaussian smoothing on gradient image
reg_params["gradient_kernel"] = 3 # kernel size for image gradient calculation
reg_params["gradient_write"] = True # write gradient image to file
reg_params["cost_step"] = 1000 # number of iterations between cost function checks
reg_params["cost_sample"] = 10 # sample size to fit cost function

Discussion

Fig. 3 demonstrates that the application of GBB on the spherical source mesh results in the right output. However, it should be noted that solely vertices are moved towards the new boundary and the mesh topology is kept the same. For example, in the current case, the mesh is stretched to fit the bigger cube. This results in a position-dependent increase in face areas which is depicted in fig. 4. To maintain homogenous sampling of data across the surface mesh, the final GM/WM boundary should be “remeshed”.
